## 1. 项目概述

本项目拟建设一个专业的中国朝代更迭地图演示系统，用真实欧亚大陆物理地图作为舞台，按年份播放中国历史政权的存在、并立、继承和消亡过程。系统第一版面向演示、教学、汇报与后续数据工程验证，优先做成可在浏览器运行的 Web 演示系统。

系统的核心体验是：用户打开页面后，看到带海拔、山脉、河流、湖泊、海岸线和主要地理标识的欧亚大陆地图；点击播放后，系统按年推进时间线，逐年展示当年存在的所有政权。若某年存在多个并立政权，则全部展示；若某年只有一个政权，则只展示一个。每个政权同时展示君王、年号、都城、创建者、末代君主、灭亡或继承关系、近似疆域面积、数据置信度与来源说明。

第一版的关键产品决策是：采用 `input/v03`（已完成 180 个政权、3.7 万行数据的全量《二十四史》等权威史籍交叉验证）。目前提供了高质量的政权、君主、年度展开和争议说明数据，但没有历史疆域 polygon 和历史精确面积字段。因此当前阶段采用“现代行政区近似”口径：从 `modern_admin_units_raw` 等字段解析现代行政区文本，匹配现代行政边界，生成近似疆域 polygon 和近似面积。所有相关 UI 和数据字段必须标注“现代行政区近似，非历史精确边界”。

## 2. 目标与非目标

### 2.1 产品目标

1. 建立一个可演示的中国历史政权年度地图播放系统，按年展示政权更迭、并立与继承关系。
2. 以欧亚大陆真实物理地图作为背景，提供海拔、山脉、河流、湖泊等地理语境，让用户理解历史政权与自然地理的关系。
3. 完整读取 `input/v03` 目录下所有核心数据，并在界面中体现数据置信度、争议说明和来源追溯。
4. 支持播放、暂停、调速、上一年、下一年、年份滑条、年份输入、搜索和筛选。
5. 每次打开应用时，自动恢复到上一次关闭前的本地状态，包括年份、播放状态、播放速度、地图视角、图层、筛选、选中政权和界面布局。
6. 为后续补充历史精确疆域、CHGIS/CCTS 等专业历史 GIS 数据预留数据接口和质量标记。

### 2.2 非目标

1. 当前版本不承诺提供历史精确疆域边界；疆域与面积仅作为现代行政区近似演示。
2. 当前版本不把 MRD 作为实现代码说明文档；具体技术方案可在后续 PRD、架构设计或开发任务中细化。
3. 当前版本不做用户账户、多人协作、在线编辑、商业授权管理。
4. 当前版本不提供跨设备自动云同步；状态恢复以本地设备和用户显式导入导出为主。
5. 当前版本不处理所有古今地名异名的人工校勘，只要求记录无法匹配、低置信度和待人工复核项。

## 3. 数据基线与约束

### 3.1 v03 数据文件

系统必须读取 `input/v03` 下的 5 个 CSV 文件和 README：

| 文件 | 行数 | 用途 |
| --- | ---: | --- |
| `chinese_history_polities_master_v03.csv` | 167 | 政权标准表。一个政权一个 `polity_id`，包含名称、起止年、类型、现代地理文本、都城、创建者、灭亡或继承关系、来源、置信度等。 |
| `chinese_history_rulers_master_v03.csv` | 934 | 君主标准表。一个君主一个 `ruler_id`，通过 `polity_id` 关联政权，包含统治起止年、年号、来源和置信度。 |
| `chinese_history_polities_yearly_v03.csv` | 36,359 | 年度展开主表。地图和时间轴必须优先使用此表，按 `year` 查询当年政权和君主。 |
| `chinese_history_unresolved_or_disputed_v03.csv` | 34 | 争议、未定与口径说明表。用于 UI 风险提示和来源说明。 |
| `chinese_history_validation_report_v03.csv` | 9 | 数据工程质检结果。用于展示 PASS/WARN 状态和验收数据完整性。 |

### 3.2 年份与时间口径

1. 年份存储沿用 v03 口径：BCE 使用负整数，CE 使用正整数，无公元 0 年。
2. 当前年度主表覆盖范围为前 1046 年至 1912 年，共 2,955 个不同年份。
3. 年份展示规则：
   - `-1046` 展示为“前1046年”。
   - `1912` 展示为“1912年”。
   - 时间轴必须跳过公元 0 年。
4. 年度播放单位为“年”，不做月、日、季节等更细粒度。

### 3.3 并存政权约束

1. 系统按年度展示当年所有存在政权。
2. 数据中单年最大并存年度行数量为 54，第一版必须以这个规模作为性能与视觉验收基线。
3. 并存政权数量较多时，地图默认全部显示，但必须提供图例、颜色分组、列表定位和搜索，以降低视觉负担。

### 3.4 数据质量约束

1. `validation_report` 中 `year_within_polity_range`、`polity_year_completeness`、`duplicate_row_id`、`yearly_polity_id_join`、`yearly_ruler_id_join`、`year_sort_order`、`no_year_zero`、`polity_sources_present` 均为 PASS，系统应在数据概览中展示该状态。
2. `partial_polity_boundaries` 为 WARN，涉及中山国、薛国。系统必须在详情面板提示存在部分边界不完整口径。
3. 不能因为君主字段为空而隐藏政权。`row_granularity = year_polity_unmatched_ruler` 表示政权存在但暂未匹配到可解析君主年表。

## 4. 用户与场景

### 4.1 核心用户

1. 历史内容演示者：需要在会议、课堂或视频脚本中快速播放中国朝代更替过程。
2. 历史学习者：希望通过地图理解政权并立、疆域变化和自然地理之间的关系。
3. 数据编辑与研究人员：需要检查 v03 数据质量、争议项、来源字段和后续历史边界补充优先级。
4. 产品与研发团队：需要用第一版演示系统验证数据结构、地图渲染和时间轴交互是否可持续演进。

### 4.2 典型场景

1. 演示者点击播放，系统从前 1046 年按设定速度推进，自动展示每年的政权分布。
2. 用户暂停在三国、五胡十六国、南北朝、五代十国等并立时期，查看当年所有政权和君王。
3. 用户输入某一年，例如 `-221` 或 `1368`，系统跳转到该年并更新地图、列表和详情。
4. 用户搜索“秦朝”“刘邦”“忽必烈”等关键词，系统定位关联年份、政权或君主。
5. 数据编辑人员打开争议过滤器，只查看有争议、低置信度或无法生成近似疆域的政权。

## 5. 同类系统调研结论

### 5.1 GeaCron

GeaCron 将自身定位为历史研究、教育和传播使用的 geo-temporal database，并强调可配置时间线、全球任意区域任意历史时期地图，以及每年年初的世界历史状态展示。对本项目的启发是：年度时间切片应成为一等数据模型，系统应围绕“某一年发生了什么”组织查询、渲染和播放，而不是只做静态朝代表。

可借鉴功能：
1. 年度切片地图。
2. 可配置时间线。
3. 全球或大区域空间视角。
4. 矢量数据和栅格输出能力。

本项目采用：
1. 以 `year` 为核心索引，生成 `years/{year}.json`。
2. 播放时按年度完整替换或过渡地图状态。
3. 预留导出当前年份截图和后续视频输出能力。

### 5.2 Running Reality

Running Reality 强调单一全球 geo-temporal model，支持 web map、桌面应用、历史数据、地点、人物、事件、引用和可验证来源。它的价值在于不仅展示边界，还把“地点、人物、事件、引用”组织成可追溯的时空模型。

可借鉴功能：
1. 动态地图与时间控制。
2. 人物、地点、事件的结构化关联。
3. 每个数据点保留开放、可追溯、可验证引用。
4. 支持嵌入和教学场景。

本项目采用：
1. 政权与君主以 `polity_id`、`ruler_id` 关联。
2. 信息面板必须展示来源和置信度。
3. 后续可扩展事件、战役、都城迁移和人物层。

### 5.3 OpenHistoricalMap

OpenHistoricalMap 在 OpenStreetMap 经验基础上加入时间维度，用户可通过 time slider 浏览地点在不同时间的状态。对本项目的启发是：历史地图数据需要时间属性和可编辑/可追溯规范，不能只是一张当前地图。

可借鉴功能：
1. 时间滑条。
2. 面向贡献和复用的数据结构。
3. 对历史对象增加时间维度。

本项目采用：
1. 年份滑条作为主控件。
2. 衍生疆域数据必须记录生成口径、来源、时间戳和置信度。
3. 后续历史精确边界可采用 `valid_from_year`、`valid_to_year` 结构扩展。

### 5.4 ArcGIS time-enabled layers

ArcGIS StoryMaps 对时间数据的建议包括：以时间属性过滤、使用 time-enabled layer、配置播放间隔和播放速度；同时指出复杂或数据量较大的时间动画可能需要预渲染或更强性能策略。

可借鉴功能：
1. 时间属性过滤。
2. 播放间隔与播放速度配置。
3. 一次只显示一个时间片或累计显示的开关。
4. 复杂时间数据的性能分层。

本项目采用：
1. 默认“一次只显示当前年份”。
2. 播放速度可设置。
3. 预处理年度 JSON，避免播放时实时扫描 36,359 行 CSV。
4. 后续可增加“累计显示”模式，用于展示长期扩张轨迹。

### 5.5 OldMapsOnline、CHGIS、CCTS 与 WHG

OldMapsOnline 代表历史图叠加和地图索引方向；CHGIS 是中国历史地理信息系统，覆盖公元前 221 年至 1911 年历史行政单位，并提供历史海岸线、主要河流、概化高程等图层；CCTS 以谭其骧《中国历史地图集》和 Arc/China 为基础，但版权限制较强；WHG 提供历史地名、几何、时间边界、权威来源和 reconciliation API。

可借鉴功能：
1. 历史图叠加与对照。
2. 历史行政单位和历史地名体系。
3. 地名 reconciliation 与外部权威 ID。
4. 历史数据授权边界管理。

本项目采用：
1. 当前版本不直接复制受限版权历史边界。
2. 以现代行政区近似作为演示方案。
3. 后续优先评估 CHGIS、公开历史边界数据或自建校勘边界。
4. 地名匹配结果必须可审计，不允许静默覆盖。

## 6. 功能需求

### 6.1 地图底图

需求编号：MAP-001  
优先级：P0  
需求描述：系统以欧亚大陆真实地图作为背景，默认视角覆盖中国及周边欧亚区域，并可缩放、平移。

验收标准：
1. 首屏可见中国及周边欧亚地理环境。
2. 地图包含海岸线、陆地、海洋和主要地理边界。
3. 支持缩放、平移、重置视角。

需求编号：MAP-002  
优先级：P0  
需求描述：底图必须包含物理地理信息，包括海拔阴影、主要山脉、主要河流、湖泊和主要地名。

推荐数据源：
1. NOAA ETOPO 2022：全球地形、海底地形和岸线数据，可用于海拔阴影或高程渲染。
2. Natural Earth physical vectors：海岸线、陆地、水体、河流、湖泊、物理标签等。
3. HydroRIVERS：全球河流矢量网络，可用于更细的河流层。

验收标准：
1. 用户可以打开或关闭海拔、山脉/物理标签、河流、湖泊层。
2. 主要河流与湖泊不遮挡政权疆域。
3. 低缩放级别展示概览，高缩放级别展示更详细地理要素。

### 6.2 时间轴与播放控制

需求编号：TIME-001  
优先级：P0  
需求描述：系统提供播放、暂停、上一年、下一年、速度设置、年份滑条和年份输入。

验收标准：
1. 点击播放后，年份按设定速度自动推进。
2. 点击暂停后，年份停止推进，地图状态保持在当前年份。
3. 上一年/下一年按钮按 v03 年份规则跳转，跨 BCE/CE 时跳过公元 0 年。
4. 年份输入支持负整数和中文展示转换。
5. 速度至少支持 0.5x、1x、2x、5x、10x。

需求编号：TIME-002  
优先级：P0  
需求描述：播放单位为“年”，系统按每一年展示当年全部政权状态。

验收标准：
1. `years/{year}.json` 是前端播放的主要数据入口。
2. 每个年份只展示当年存在的政权，不累计展示过去政权。
3. 后续可扩展“累计轨迹”模式，但当前版本默认关闭。

### 6.3 年度政权展示

需求编号：POLITY-001  
优先级：P0  
需求描述：每个年份展示所有存在政权。

验收标准：
1. 若当年存在多个政权，则地图和列表全部显示。
2. 若当年只有一个政权，则地图和列表只显示该政权。
3. 并存政权过多时，列表可滚动，地图颜色和标签仍可辨认。
4. **LOD (Level of Detail) 视距聚合策略**：在极端并存期（如春秋战国并存 54 国），地图应根据当前缩放级别（Zoom Level）进行标签防碰撞处理；低缩放级别仅显示核心大国，高缩放级别逐步展示小国标签。

需求编号：POLITY-002  
优先级：P0  
需求描述：政权颜色按宏观时期或朝代分组生成，并根据政权类型（游牧/定居）实施差异化视觉渲染。

验收标准：
1. 同一 `polity_id` 在所有年份保持同一主色。
2. 同一 `macro_period` 或 `dynasty_name` 使用接近但可区分的色系。
3. 高亮政权与普通政权视觉差异明显。
4. **游牧政权差异化渲染**：对于 `polity_type` 为纯游牧政权（如高车、柔然、丁零），其疆域应渲染为**虚线边界 + 内侧羽化渐变（Glow/Gradient）或斑马纹理**，以表达“活动势力范围”，与中原定居王朝的实线纯色填充形成明确区分。

### 6.4 疆域与面积

需求编号：TERR-001  
优先级：P0  
需求描述：当前版本使用现代行政区近似疆域。系统从 `modern_admin_units_raw` 解析行政区文本，匹配现代行政边界，生成近似 polygon。

验收标准：
1. 成功匹配行政区的政权在地图上显示近似疆域。
2. 匹配失败或字段缺失时，不显示伪造疆域，改为显示“待补充/无法估算”。
3. 地图图例、详情面板和导出数据中必须标注“现代行政区近似，非历史精确边界”。

需求编号：TERR-002  
优先级：P0  
需求描述：系统计算并展示近似面积。

验收标准：
1. 面积单位为平方公里。
2. 面积来源为 polygon geodesic area 计算结果，原始工具可采用 Turf.js `area`。
3. 面积字段命名为 `approx_area_km2`，不得命名为 `historical_area_km2`。
4. 无 polygon 时面积为空，并展示“无法估算”。

需求编号：TERR-003  
优先级：P1  
需求描述：近似疆域数据必须可审计。

验收标准：
1. 每条近似疆域记录包含 `polity_id`、`source_text`、`matched_admin_units`、`match_confidence`、`geometry_source`、`generated_at`。
2. 匹配置信度低于阈值时在 UI 中显示警示。
3. 允许人工维护 override 表修正匹配结果。

### 6.5 信息面板

需求编号：INFO-001  
优先级：P0  
需求描述：当用户点击政权 polygon、列表项或搜索结果时，展示政权详情。

必须展示字段：
1. 年份与年份标签。
2. 政权名称、别名、宏观时期、朝代分组、政权类型。
3. 政权起止年、都城（支持**多都城动态化**：如某年发生迁都，面板与地图应实时体现最新都城）、族属或统治家族。
4. 创建者、末代君主、灭亡或继承关系。
5. 当前年份君主、称号、庙号、谥号、本名、统治起止年、年号。
6. 近似面积、近似疆域口径说明。
7. **数据溯源（极高优）**：政权置信度、**考证文献出处（`polity_source_titles`）**、**史料原文说明（`polity_source_raw`）**、君主置信度、来源 URL。这是体现本系统权威性的核心板块。
8. 争议或未定说明。

验收标准：
1. 君主缺失时显示“该年度暂未匹配到可解析君主年表”，不得显示为空白错误。
2. 来源 URL 可点击，史料原文说明必须醒目展示。
3. 争议项以明显但不惊扰的提示样式呈现。
4. 当播放到政权迁都年份时，都城字段与地图上的都城 Marker 必须同步更新转移。

### 6.6 搜索与筛选

需求编号：SEARCH-001  
优先级：P1  
需求描述：支持按年份、政权、君主搜索，并引入“别名/同义词映射库”。

验收标准：
1. 输入年份可直接跳转。
2. 输入政权名可返回关联政权和存在年份范围。支持**异名同国及简繁同体合并搜索**（例如：搜索“刘宋”、“宋朝”、“南朝宋”，或搜索“后唐”、“後唐”，系统均能精准定位到同一个对应的政权实体）。
3. 输入君主名可返回君主所属政权和统治年份范围。

需求编号：FILTER-001  
优先级：P1  
需求描述：支持筛选政权类型、低置信度项、争议项和疆域状态。

验收标准：
1. 可隐藏或显示诸侯国/古国、王朝/政权、割据政权等类型。
2. 可只显示 `confidence_score` 低于指定阈值的政权。
3. 可只显示有 unresolved/disputed 记录的政权。
4. 可只显示“已生成近似疆域”或“无法估算疆域”的政权。

### 6.7 演示体验

需求编号：DEMO-001  
优先级：P1  
需求描述：系统提供适合历史变迁视频风格的演示体验，强化“因果变动”解释力。

验收标准：
1. 播放时疆域变化使用淡入淡出或平滑过渡。
2. 当前年份的关键政权在地图或列表中有高亮。
3. **关键历史事件节点层（Events Layer）**：当时间线推进到导致疆域巨变或政权更替的特定年份时，系统应展示轻量化的事件气泡或提示（例如：“前221年 秦灭齐，统一六国”、“383年 淝水之战”、“1127年 靖康之变”），增强政权演变的因果解释力。
3. 时间线对重要节点显示标记，例如秦统一、三国并立、隋唐交替、宋辽金夏、元明清交替。
4. 支持自动镜头跟随当前主要政权范围；并存数量多时保持稳定视角。
5. 支持一键重置到默认欧亚视角。

### 6.8 本地状态保持

需求编号：STATE-001  
优先级：P0  
需求描述：应用每次打开时，必须自动恢复到上一次关闭前的本地状态。恢复范围包括当前年份、播放/暂停状态、播放速度、地图视角、选中政权、打开的信息面板、图层开关、筛选条件、搜索关键词、时间线模式和界面布局。

验收标准：
1. 用户关闭应用或浏览器标签页后再次打开，系统自动恢复到关闭前的年份、地图视角、图层、筛选和选中项。
2. 若关闭前处于播放状态，重新打开并完成数据加载后，应按原播放速度继续播放；若恢复失败，必须提示原因并停留在已恢复年份。
3. 若关闭前处于暂停状态，重新打开后保持暂停状态。
4. 状态写入不能明显影响播放性能，播放时不得因高频状态保存造成卡顿。
5. 用户可以一键恢复默认状态，清除本地保存的应用状态。

需求编号：STATE-002  
优先级：P0  
需求描述：本地状态保持机制必须具备良好的可移植性和扩展性，不能与单一浏览器、单一数据版本或单一部署路径强绑定。

验收标准：
1. 状态数据使用版本化 JSON schema，包含 `schema_version`、`app_version`、`data_version`、`updated_at`。
2. 状态中不得保存绝对文件路径、临时 blob URL 或浏览器私有对象；所有引用必须使用稳定业务 ID，例如 `polity_id`、`ruler_id`、年份、图层 ID。
3. 系统必须提供“导出状态”和“导入状态”能力，导出的 JSON 文件可在另一台设备或另一浏览器中恢复。
4. 状态模型必须支持迁移。当应用版本或数据版本变化时，系统优先保留可迁移字段，并清除或降级不可用字段。
5. 存储实现必须通过统一 `StateStore` 抽象访问，后续可替换为 IndexedDB、localStorage、桌面文件存储或服务端同步。

需求编号：STATE-003  
优先级：P1  
需求描述：状态恢复必须可靠处理异常情况。

验收标准：
1. 本地状态损坏、字段缺失或 schema 不兼容时，系统回退到默认状态，并提示用户“已重置损坏状态”。
2. 状态中的年份超出当前数据版本范围时，跳转到最近可用年份。
3. 状态中的 `polity_id` 或 `ruler_id` 在当前数据版本不存在时，清除选中项，但保留年份、图层、筛选和地图视角。
4. 多标签页同时打开时，采用 `updated_at` 较新的状态作为最终状态，并通过跨标签同步避免互相覆盖。
5. 用户导入状态前，系统必须校验 schema、数据版本和年份范围。

## 7. 数据处理与接口需求

### 7.1 预处理流程

系统应提供离线预处理脚本，将 CSV 转换为前端可快速加载的数据资产。

输入：
1. `input/v03/chinese_history_polities_master_v03.csv`
2. `input/v03/chinese_history_rulers_master_v03.csv`
3. `input/v03/chinese_history_polities_yearly_v03.csv`
4. `input/v03/chinese_history_unresolved_or_disputed_v03.csv`
5. `input/v03/chinese_history_validation_report_v03.csv`
6. 现代行政边界数据。
7. 物理地理底图数据。

输出：
1. `public/data/v03/metadata.json`
2. `public/data/v03/polities.json`
3. `public/data/v03/rulers.json`
4. `public/data/v03/issues.json`
5. `public/data/v03/validation.json`
6. `public/data/v03/years/{year}.json`
7. `public/data/v03/territories/approx_polities.geojson` 或 PMTiles
8. `public/data/v03/territories/territory_match_report.csv`

### 7.2 年度数据接口

`years/{year}.json` 建议结构：

```json
{
  "year": -221,
  "year_label": "前221年",
  "polity_count": 1,
  "polities": [
    {
      "polity_id": "polity_xxxx",
      "polity_name": "秦朝",
      "macro_period": "秦汉",
      "dynasty_name": "秦朝",
      "polity_type": "王朝/政权",
      "rulers": [
        {
          "ruler_id": "ruler_xxxxx",
          "ruler_name": "秦始皇",
          "era_names": "",
          "ruler_reign_start_year": -246,
          "ruler_reign_end_year": -210
        }
      ],
      "territory": {
        "geometry_ref": "approx_polities.geojson#polity_xxxx",
        "territory_method": "modern_admin_approximation",
        "approx_area_km2": 0,
        "match_confidence": 0,
        "label": "现代行政区近似，非历史精确边界"
      },
      "quality": {
        "confidence_score": 90,
        "has_dispute": false,
        "has_unmatched_ruler": false
      }
    }
  ]
}
```

说明：
1. `approx_area_km2` 示例值为占位，实际生成时必须由 polygon 计算。
2. 一个年度政权可能匹配多个 ruler 行，前端应支持展示多位君主或统治交接。
3. `geometry_ref` 可在 GeoJSON、PMTiles 或矢量瓦片实现中调整，但必须可追溯到 `polity_id`。

### 7.3 主键与兼容性

1. 所有跨表关联必须使用 `polity_id` 和 `ruler_id`。
2. 禁止依赖 CSV 行号和 `row_id` 作为长期业务主键。
3. 预处理输出必须记录 `data_version = "v03"` 和生成时间。
4. 后续 v04 数据进入时，系统应允许并行保留多个数据版本。

### 7.4 本地状态数据模型

本地状态是用户在应用中的操作现场，不是 v03 历史数据本身。系统不得把年度 JSON、GeoJSON、底图瓦片等大体积数据重复写入状态存储，只保存可恢复现场所需的轻量引用和配置。

推荐状态结构：

```json
{
  "schema_version": "1.0.0",
  "app_id": "chinese-dynasty-history-map",
  "app_version": "0.1.1",
  "data_version": "v03",
  "updated_at": "2026-05-15T12:00:00Z",
  "timeline": {
    "current_year": -221,
    "is_playing": false,
    "playback_speed": 2,
    "timeline_mode": "current_year"
  },
  "map_view": {
    "center": [105.0, 35.0],
    "zoom": 3.8,
    "bearing": 0,
    "pitch": 0,
    "auto_follow": true
  },
  "selection": {
    "selected_polity_id": "polity_xxxx",
    "selected_ruler_id": "ruler_xxxxx",
    "active_panel": "polity_detail"
  },
  "layers": {
    "terrain": true,
    "mountains": true,
    "rivers": true,
    "lakes": true,
    "physical_labels": true,
    "territories": true
  },
  "filters": {
    "polity_types": ["诸侯国/古国", "王朝/政权", "政权/国家", "割据政权/国家", "王朝/割据政权"],
    "min_confidence_score": 0,
    "show_disputed": true,
    "show_unmatched_ruler": true,
    "territory_status": "all",
    "search_keyword": ""
  },
  "ui": {
    "sidebar_open": true,
    "active_tab": "year_overview",
    "collapsed_sections": [],
    "theme": "system"
  }
}
```

实现要求：
1. `timeline.current_year` 必须经过当前数据版本年份范围校验。
2. `selection` 中所有 ID 必须在当前数据索引中存在，不存在则清除。
3. `layers` 和 `filters` 必须允许未来新增字段，未知字段不得导致恢复失败。
4. `updated_at` 使用 ISO 8601 UTC 时间，用于多标签页冲突处理和导入状态新旧判断。
5. `schema_version` 变化时必须通过 migration registry 迁移，不能直接丢弃全部状态，除非状态损坏不可解析。

### 7.5 StateStore 抽象与存储适配

系统必须通过统一 `StateStore` 访问本地状态，避免业务组件直接读写浏览器存储。

`StateStore` 必须提供能力：
1. `load()`：加载、校验、迁移并返回可用状态。
2. `save(state)`：保存完整状态。
3. `patch(partialState)`：局部更新状态。
4. `subscribe(listener)`：订阅状态变化，用于 UI 同步。
5. `export()`：导出可移植 JSON。
6. `import(json)`：校验并导入外部状态。
7. `reset()`：清除本地状态并恢复默认值。

存储适配策略：
1. Web 端主存储使用 IndexedDB，适合结构化、版本化和较大状态扩展。
2. localStorage 仅作为兼容性降级和启动索引，不作为复杂状态主存储。
3. 若未来封装桌面应用，可增加文件存储适配器，将同一 JSON schema 保存为本地状态文件。
4. 若未来增加账号系统或云同步，可增加远程同步适配器，但不得改变前端业务层调用接口。

写入策略：
1. 用户主动操作后 debounce 保存，建议 300ms 至 1000ms。
2. 年份播放推进时只在年度切换完成后保存当前年份，不按动画帧保存。
3. `beforeunload`、页面隐藏、应用关闭前必须尽力 flush 最新状态。
4. 状态保存失败时必须记录错误，并在 UI 中提供非阻塞提示。

## 8. 推荐技术方向

### 8.1 前端地图

推荐使用 MapLibre GL JS 作为底图和矢量瓦片渲染基础，deck.gl 作为大规模 polygon、标签、动画和交互增强层。原因是 MapLibre 支持 WebGL 地图、样式规范、GeoJSONSource、VectorTileSource 和 RasterDEMTileSource 等能力；deck.gl 适合做高性能数据层。

### 8.2 数据资产

1. 小规模验证期可使用静态 JSON + GeoJSON。
2. 疆域和底图变大后，应转换为 PMTiles 或矢量瓦片。
3. 年度索引必须静态化，避免运行时扫描 CSV。
4. CSV 原始文件保留在 `input/v03`，前端只读取预处理后的公开数据。

### 8.3 面积计算

近似面积推荐使用 Turf.js `area` 计算 polygon geodesic area，单位从平方米转换为平方公里。面积字段必须与匹配置信度、疆域口径一起显示。

### 8.4 本地状态技术方向

本地状态模块建议采用 local-first 设计。应用启动时先加载静态数据 metadata，再加载本地状态，完成 schema 迁移和数据版本校验后恢复 UI。若本地状态可用，则以本地状态为准；若不可用，则使用系统默认状态。

技术建议：
1. 使用 IndexedDB 保存主状态，object store 建议命名为 `app_state`，主键为 `state_id = "last_session"`。
2. 使用 localStorage 保存极简启动标记，例如 `last_state_updated_at` 和 `last_state_schema_version`，帮助快速判断是否存在可恢复状态。
3. 使用 BroadcastChannel 同步多标签页状态变更，避免一个标签页修改筛选条件后另一个标签页覆盖旧状态。
4. 使用 JSON schema 或同等校验逻辑验证导入状态和本地状态。
5. 对外导出的状态文件建议命名为 `history-map-state-{data_version}-{yyyyMMdd-HHmmss}.json`。

## 9. UI 信息架构

### 9.1 主界面

1. 地图画布：占据主要区域，显示物理底图、政权疆域、标签和高亮状态。
2. 顶部或底部时间控制条：年份、播放/暂停、上一年/下一年、速度、滑条。
3. 右侧信息面板：显示当前年份概况、政权列表、选中政权详情。
4. 左侧或浮层工具：图层开关、搜索、筛选、图例。
5. 状态管理入口：提供恢复提示、导出状态、导入状态、重置为默认状态。
6. 来源与质量浮层：显示数据版本、校验状态、争议说明和外部底图来源。

### 9.2 关键状态

1. Loading：加载数据与地图资产。
2. Ready：地图和当前年份数据可用。
3. Playing：自动播放中。
4. Paused：暂停并允许查看详情。
5. Empty Territory：政权存在但无近似疆域。
6. Data Warning：存在争议、低置信度或部分边界不完整。
7. Restoring：正在加载本地状态并恢复上次关闭现场。
8. State Warning：本地状态损坏、过期、不兼容或部分字段无法恢复。
9. Error：数据文件缺失、年度 JSON 无法加载或地图资产失败。


### 9.3 移动端与响应式降级策略

面对平板和大屏手机访问的历史学习者，主界面必须具备合理的降级体验。

1. **布局响应**：屏幕宽度小于 768px 时，右侧信息面板转为底部抽屉式弹窗（Bottom Sheet Drawer），时间轴控制条自动精简为紧凑模式。
2. **交互降级**：地图上的 Hover 交互统一降级为 Click 触发；并存政权列表提供更友好的触屏滚动区。

## 10. 非功能需求

### 10.1 性能

1. 首屏应在合理网络环境下尽快展示底图和当前年份，不应等待全部年度 JSON 加载完成。
2. 播放时每次年度切换目标响应时间不超过 200ms，地图过渡动画不明显卡顿。
3. 最大并存政权年度行数量为 54，必须作为基础压力场景。
4. 年度 JSON 应按需加载并缓存相邻年份。

### 10.2 可用性

1. 时间控制必须适合演示场景，按钮大而清晰。
2. 年份展示必须避免 BCE/CE 混淆。
3. 信息面板不应被地图遮挡。
4. 并存政权过多时，列表和图例必须可扫描。

### 10.3 准确性与可信度

1. 任何近似数据都必须显式标注。
2. 来源链接必须可追溯。
3. 争议项不得被隐藏为确定事实。
4. 系统必须允许后续接入更高质量历史边界替换现代行政近似。

### 10.4 可维护性

1. 预处理脚本应可重复运行。
2. 生成结果应包含匹配报告，便于人工复核。
3. 数据版本、生成时间、输入文件 hash 或 mtime 应写入 metadata。
4. 前端组件应区分地图层、时间控制、详情面板、数据加载和筛选逻辑。

### 10.5 本地状态可移植性与扩展性

1. 本地状态 schema 必须独立于具体存储实现，保证 Web、桌面封装和未来云同步可以复用同一状态模型。
2. 状态文件不得包含绝对路径、运行时临时 URL、浏览器对象或不可序列化字段。
3. 导出状态必须是可读 JSON，便于人工检查、版本迁移和跨设备迁移。
4. 状态迁移必须向后兼容至少一个小版本；无法迁移时保留原始状态备份并提示用户。
5. 状态模块必须有单元测试覆盖默认状态、保存、恢复、导入、导出、迁移、损坏回退和多标签冲突处理。

2. 君主来源标题与 URL 可查看。
3. 外部地图、物理地理、河流、面积计算和专业系统参考均在“来源与参考”中列出。
4. 争议说明可从 `unresolved_or_disputed` 追溯。

### 11.6 本地状态验收

1. 用户跳转到任意年份、调整地图视角、打开图层、设置筛选、选中政权后关闭应用，再次打开时完整恢复。
2. 用户关闭前处于播放状态，再次打开后按原播放速度继续播放。
3. 用户关闭前处于暂停状态，再次打开后保持暂停并恢复当前详情面板。
4. 导出状态 JSON 后，在另一浏览器或另一台设备导入，可恢复年份、视角、图层、筛选和选中项。
5. 数据版本变化后，系统保留可迁移状态，清除不存在的 `polity_id` 或 `ruler_id`，并给出提示。
6. 本地状态损坏时，系统自动回退默认状态，不影响应用启动。
7. 多标签页同时操作时，以最新 `updated_at` 状态为准，且不会出现无限互相覆盖。

## 12. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 现代行政区近似与历史疆域差异较大 | 用户误以为是精确历史边界 | UI、数据字段、图例、导出均明确标注“近似”；后续接入历史边界数据。 |
| `modern_admin_units_raw` 缺失较多 | 部分政权无法生成 polygon 与面积 | 展示“待补充/无法估算”；输出匹配报告，形成数据补全任务。 |
| 周代诸侯国并存数量大 | 地图视觉拥挤 | 全部显示但提供颜色分组、列表定位、搜索、自动聚焦和可调标签密度。 |
| 历史边界数据版权复杂 | 无法直接使用专业历史地图集 | 第一版不复制受限数据；只调研 CHGIS/CCTS 等方向，并记录授权边界。 |
| 实时渲染大量地理数据卡顿 | 播放体验下降 | 年度 JSON 预生成、相邻年份缓存、疆域矢量瓦片化、必要时简化几何。 |
| 君主年度匹配为空 | 用户误判数据缺失或政权不存在 | 按 v03 说明展示 unmatched ruler 状态，不隐藏政权。 |
| 本地状态损坏或 schema 过期 | 应用无法恢复或启动异常 | 启动时校验 schema，失败则备份损坏状态并回退默认状态。 |
| 多标签页同时写入状态 | 上次状态被旧标签覆盖 | 使用 `updated_at`、BroadcastChannel 和 last-write-wins 策略。 |
| 状态导出后跨设备不可用 | 用户无法迁移演示现场 | 状态只保存稳定业务 ID 和可序列化 JSON，不保存绝对路径或临时对象。 |

## 15. 关键决策摘要

1. 第一版做 Web 演示系统。
2. 播放粒度为年。
3. 每一年展示当年全部存在政权。
4. 当前版本疆域采用现代行政区近似。
5. 面积只展示近似值，字段命名为 `approx_area_km2`。
6. 数据主键使用 `polity_id` 和 `ruler_id`。
7. CSV 只作为输入，前端使用预处理后的静态 JSON 与 GeoJSON/PMTiles。
8. 争议、低置信度、缺失君主、缺失疆域必须显式展示。
9. CHGIS、CCTS、WHG 等专业系统作为后续历史精确化方向，不在当前版本直接作为边界数据来源。
10. 应用必须每次打开恢复上一次关闭状态，本地状态采用版本化 JSON schema 和 `StateStore` 抽象。
11. 本地状态以 IndexedDB 为 Web 主存储，并支持 JSON 导入导出，以保证可移植性和后续扩展。
